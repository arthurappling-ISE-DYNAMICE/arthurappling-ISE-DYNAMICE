## I. DAILY EXECUTION WORKFLOW (THE 5-STEP PROTOCOL)

To maintain absolute structural consistency, every execution cycle must follow these five sequential phases.

```
[1. AUDIT & INGEST] ──► [2. BOTTLENECK ANALYSIS] ──► [3. SYSTEM MANUFACTURING] ──► [4. CREDIT AUDIT] ──► [5. COMPLIANCE & PUSH]
```

### Phase 1: Audit & Ingest
1. Review the existing `/temporary/MASTER_EXECUTION_LEDGER.csv` (or equivalent execution logs) to ensure no duplicate work is performed.
2. Ingest pre-existing assets (Regional Lead Databases, Friction Audits, and Funding Intelligence) to establish a baseline.
3. Check the current sandbox or system state to verify environment readiness.

### Phase 2: Bottleneck Analysis
Identify the highest-leverage operational bottleneck among target NorCal industries (logistics, hauling, industrial painting). Determine if the barrier is:
* **Capital Access:** Lack of structured compliance for BAAQMD VIP or Workforce Stability Grants.
* **Conversion Rate:** Friction in client onboarding or booking systems.
* **Operational Leakage:** Missing standard operating procedures (SOPs) or manual reporting overhead.

### Phase 3: System Manufacturing
Develop an implementation-ready asset to eliminate the bottleneck. Use the standardized **System Manufacturing Template** (Section III of this document) to structure the asset.

### Phase 4: Credit Audit
Evaluate the computational efficiency of the build. Verify that high-leverage AI credits were spent strictly on structural code, advanced systems design, or institutional-grade documentation, rather than redundant web scraping or generic summaries.

### Phase 5: Compliance & Push
Validate the asset against the **Validation Contract**. Push the verified asset to the `arthurappling-ISE-DYNAMICE` repository under its designated WAT directory.

## IV. VALIDATION & DEPLOYMENT PROTOCOL

Every asset must be audited before pushing to the remote repository.

### 1. The Validation Contract
No system is considered complete without an explicit, executable test path.
* **Exact Command:** The literal command that must be executed in the sandbox or terminal to run the system.
* **Pass Criteria:** A precise, objective description of the success state (e.g., a specific file created with exact columns, or a local server running on a specific port).
* **Error Map:** A troubleshooting matrix mapping the most likely failure modes to their exact fixes.

### 2. Zero-Inference Rule
If a script or system fails its validation contract twice during testing, **HALT execution immediately**. Do not attempt to guess the fix. Generate a **Ground Truth Audit** (located in `/temporary/GROUND_TRUTH_AUDIT.md`) containing the exact error logs, environment state, and system screenshots, then await manual inspection.
